# Swift Supervisor Rewrite (Issue #86)

Goal: replace the current Node.js supervisor (`wrapper/src/run.js`) with a **native macOS Swift executable** that owns the security boundary (Keychain ACL + liveness) and launches the FUSE daemon + gateway with a minimal, verified environment.

This document is the **implementation plan + interface contract** so we can migrate incrementally without breaking existing workflows.

## Non-goals (for the first migration)

- Rewriting the FUSE daemon (that is Issue #87).
- Changing on-disk encryption formats.
- Changing default mount paths or workspace policy.

## Current behavior (baseline)

Today the supervisor:

- Creates a **liveness UNIX socket** (`OCPROTECTFS_LIVENESS_SOCK`) and passes it to child processes.
- Retrieves/creates the 32-byte **KEK** from macOS Keychain (`service=ocprotectfs`, `account=kek`).
- Passes the KEK to the FUSE daemon via an **anonymous pipe FD** (`--kek-fd <n>`).
- Starts the gateway only after a successful mount.

The FUSE daemon enforces:

- Fail-closed semantics for encrypted-path operations unless the liveness socket exists and is a UNIX socket.

## Target architecture

A single Swift executable, tentatively: `ocprotectfs-supervisor`.

Responsibilities:

1. **Keychain boundary**
   - Create/read the KEK.
   - Enforce Keychain ACL so only the expected binary identity can access it.
   - Prefer Secure Enclave / non-exportable wrapping later (Issue #78); for now keep the existing KEK storage model.

2. **Process supervision**
   - Create liveness socket.
   - Launch `ocprotectfs-fuse` with:
     - `OCPROTECTFS_LIVENESS_SOCK=<path>`
     - KEK provided via FD (pipe)
     - a *small allow-listed environment*
   - Launch gateway only after mount success.
   - On shutdown: best-effort unmount + cleanup.

3. **Testability**
   - Provide a `--dry-run` mode that exercises:
     - liveness socket creation
     - keychain access
     - FD wiring (without performing a real mount)
   - Provide deterministic logs suitable for CI.

## CLI contract (v0)

To keep migration simple, the Swift supervisor should mirror the Node wrapper flags we rely on operationally.

Proposed minimal flags:

- `--liveness-sock <path>` (or autogenerated when omitted)
- `--mountpoint <path>`
- `--backstore <path>`
- `--plaintext-prefix <path>` (repeatable)
- `--fuse-bin <path>` (defaults to `./fusefs/ocprotectfs-fuse.js` during transition)
- `--gateway-cmd <cmd...>` (optional; defaults to `openclaw gateway start` or the repo’s gateway entrypoint)
- `--dry-run`
- `--print-kek-fingerprint` (for debugging: prints SHA-256 of KEK only; never prints the KEK)

Environment variables passed to children must be allow-listed (mirroring `buildChildEnv()`).

## Keychain details

- Service: `ocprotectfs`
- Account: `kek`
- Data: 32 random bytes

Swift APIs:

- `SecItemCopyMatching` / `SecItemAdd` / `SecItemUpdate`

ACL direction:

- Phase 1: ensure the item is **not globally readable** and requires user presence when configured.
- Phase 2: bind to code signature / designated requirement.

We should document the intended ACL policy and provide a recovery/uninstall path (see Issue #89).

## Migration plan

1. **Land the interface + design doc** (this PR)
2. Add Swift package skeleton and `--dry-run` with Keychain read/create (no mounting)
3. Run Swift supervisor in acceptance tests behind a feature flag
4. Make Swift supervisor the default entrypoint
5. Remove Node supervisor

## Acceptance criteria

- `npm test` remains green.
- Operator guide updated to reference `ocprotectfs-supervisor` once it is runnable.
- Swift supervisor can be built locally on macOS with a single command.

## Security notes

- Never log or print the KEK.
- Prefer “fail closed”: if Keychain access fails or liveness socket cannot be created, do not start children.
